Protocol Benchmark Platform

A single Go binary that measures latency, throughput, and reliability of 11 network transports under identical load conditions. Server and clients run in the same process — no external infrastructure needed.

Execution Flow — Step by Step

Exactly what happens when you run go run ./cmd/benchmark-runner run --protocol tcp --scenario A

1

Blank imports fire init() for all protocols

Before main() runs, Go executes each imported package's init(). All 11 protocol packages are imported in cmd/benchmark-runner/transports.go.

// cmd/benchmark-runner/transports.go
import (
    _ "github.com/.../internal/tcp"     // init() → scenarios.Register("tcp", factory)
    _ "github.com/.../internal/udp"     // init() → scenarios.Register("udp", factory)
    _ "github.com/.../internal/http1"
    // ... 8 more
)
2

cobra parses flags → runCmd() creates ScenarioConfig

Flag values (--protocol tcp, --receivers 10, etc.) populate a ScenarioConfig struct, which is passed to NewRunner().

cfg := scenarios.ScenarioConfig{
    Protocol:      "tcp",
    Scenario:      "A",
    Duration:      30 * time.Second,
    ReceiverCount: 10,
    MsgSize:       1024,
    // ... RecvHandler is set in Step 3
}
3

Runner injects RecvHandler into config

Before looking up the factory, the runner injects its own callback. This is how client-side receives get routed back into the runner's recorder — the factory passes it to every client it creates.

// internal/scenarios/runner.go:Run()
r.cfg.RecvHandler = func(_ ConnID, data []byte, recvAt time.Time) {
    seq, sendNs, _, err := wire.DecodeHeader(data)
    if err != nil { return }
    r.recorder.RecordRecv(seq, sendNs, len(data), recvAt.UnixNano())
}
4

Factory creates Combined Transport

The factory looks up factories["tcp"] and calls it. The factory creates a wrapper struct holding both a server and a slice for N clients.

// internal/tcp/register.go
scenarios.Register("tcp", func(cfg ScenarioConfig, log *zap.Logger) (transport.Transport, error) {
    addr := fmt.Sprintf("%s:%d", cfg.ServerAddr, cfg.ServerPort)
    srv := NewTCPServer(addr, ...)
    return &tcpBenchTransport{server: srv, cfg: cfg}, nil
})
5

CombinedTransport.Start() — server binds, clients connect, wait for all

The server starts and binds the port. N clients dial in. waitForConns() polls server.Connections() until all N are visible in the server's registry (critical for async protocols like HTTP/1, HTTP/2, SSE, HTTP/3).

srv.Start()           // binds port 9000
for i := 0; i < N; i++ {
    c := NewTCPClient(WithRecvHandler(cfg.RecvHandler))
    c.Dial(addr)
    clients = append(clients, c)
}
waitForConns(srv, N)  // polls srv.Connections() ≥ N, timeout 10s
6

broadcastLoop — warmup then measurement

The runner drives a tight loop: generate → broadcast → record send. After the warmup duration the recorder is reset. Only the measurement phase data is kept.

// Runs for WarmupDuration (discarded), then Duration (measured)
for {
    data := gen()                 // e.g. 1024 random bytes
    srv.Broadcast(data)           // server wire-encodes + fans out
    n := sent.Add(1)
    if !warmup { recorder.RecordSend(n, len(data)) }
}
7

Server wire-encodes and fans out to N clients

Every server's Broadcast() creates a single wire frame (once, not per-client) and enqueues it to all client write channels. The timestamp SendNs embedded in the frame is what enables latency measurement.

// internal/tcp/server.go  (same pattern in all 11 servers)
func (s *TCPServer) Broadcast(data []byte) error {
    seq   := s.seqCounter.Add(1)
    frame := wire.Encode(seq, time.Now().UnixNano(), data)  // 24B header + data
    sized := prependLength(frame)                           // 4B len + frame
    for _, peer := range s.registry.Snapshot() {
        peer.writeCh <- sized                              // non-blocking, drops if full
    }
    return nil
}
8

Client readPump → RecvHandler → Recorder

Each client's background goroutine reads the wire frame, calls the injected handler. The handler extracts SeqNum and SendNs from the wire header to compute latency = recvNs - sendNs.

// handler injected in Step 3
func(id ConnID, wireFrame []byte, recvAt time.Time) {
    seq, sendNs, _, _ := wire.DecodeHeader(wireFrame)
    // latency = recvAt.UnixNano() - sendNs  (same-machine clock, no NTP needed)
    recorder.RecordRecv(seq, sendNs, len(wireFrame), recvAt.UnixNano())
}
9

Snapshot → RunResult → output

After the measurement phase, the recorder computes percentiles from the HDR histogram, finalises loss/reorder from the sequence tracker, and aggregates resource usage. The result is serialised to JSON, Markdown, or HTML.

snap    := recorder.Snapshot()    // latency P50/P99/P999, MsgPerSec, CPU, mem…
stats   := srv.Stats()            // lost (write-channel overflow)
result  := &collector.RunResult{
    LatP99Ns:  snap.Latency.P99.Nanoseconds(),
    MsgsPerSec: snap.MsgPerSec,
    LossRatePct: float64(stats.Lost) / float64(snap.MsgSent) * 100,
    // ...
}

Wire Protocol — 24-Byte Frame

Every benchmark message, regardless of transport, is prefixed with a fixed 24-byte little-endian header. This enables cross-protocol latency measurement, sequence-based loss/reorder detection, and message correlation between sender and receiver.

4 – 11
SeqNum
uint64 LE

SendNs
int64 LE
20 – 23

Field meanings

Magic (0xBEEFCAFE)Frame boundary marker. Validated on decode — fails fast on misalignment.
SeqNumPer-server monotonic counter. Used by SeqTracker to detect loss, reorder, duplicates.
SendNstime.Now().UnixNano() at broadcast time. Latency = recvNs − sendNs. No NTP needed (same machine).
PayloadLenByte count of payload. Bounds-checks the decode without reading payload.
PayloadRaw bytes from generator (random / sequential / JSON / binary / market tick).

Framing per protocol

TCP / HTTP/1 / HTTP/2[4B len][24B hdr + payload]
UDP[24B hdr + payload] (one datagram)
WebSocket (3 variants)WS binary frame [24B hdr + payload]
HTTP/3 stream mode[4B len][24B hdr + payload] over QUIC
HTTP/3 datagram/unistream[24B hdr + payload] direct
SSEbase64([24B hdr + payload]) as SSE data: field
WebTransport[24B hdr + payload] via QUIC stream/datagram
WebRTC[24B hdr + payload] via DataChannel

Encoding/decoding is in pkg/wire/frame.go. Little-endian throughout (no byte-swap on x86/ARM). The encode buffer is pool-recycled via sync.Pool to minimise GC in the broadcast hot path.

Factory Registration Pattern

The benchmark runner is protocol-agnostic. Protocols self-register by calling scenarios.Register() from init(). The runner never imports any protocol package directly — it imports them all as blank imports to trigger registration.

1 · Blank import

cmd/benchmark-runner/transports.go

  _ ".../internal/tcp"
  _ ".../internal/udp"
  _ ".../internal/http1"

)

Combined Transport — Server + Embedded Clients

Each factory creates a combined transport: a struct that owns both a server instance and a slice of client instances. This lets the benchmark runner measure end-to-end performance without launching separate processes.

Lifecycle

Start() {
    server.Start()
    // HTTP servers: 20ms grace for listen goroutine
    time.Sleep(20ms)

    for i := 0; i < ReceiverCount; i++ {
        c := NewClient(RecvHandler: cfg.RecvHandler)
        c.Connect(ctx)  // blocking for most protocols
        clients = append(clients, c)
    }

    // Wait until server sees all connections.
    // Critical for HTTP/1, HTTP/2, SSE, HTTP/3
    // where stream registration is async.
    waitForConns(server, ReceiverCount)
}

Stop() {
    for _, c := range clients { c.Close() }
    server.Stop()
}

Broadcast(data) → server.Broadcast(data)

RecvHandler injection

This is the key wiring that makes metrics work. The runner injects a callback before calling the factory. The factory stores it in each client. When clients receive a wire frame, they call the handler, which routes the measurement back into the runner's recorder.

// Runner injects (runner.go)
cfg.RecvHandler = func(id, data, recvAt) {
    seq, sendNs, _, err := wire.DecodeHeader(data)
    if err == nil {
        recorder.RecordRecv(seq, sendNs,
            len(data), recvAt.UnixNano())
    }
}

// Factory wires it into each client (register.go)
c := NewTCPClient(
    WithRecvHandler(cfg.RecvHandler), // ← the same fn
)

waitForConns

Replaces guessed fixed sleeps. Polls server.Connections() every 10ms until count is reached or 10–15s timeout. Prevents broadcasts from starting before clients are fully registered — especially important for SSE (stream goroutine is async) and HTTP/3 (Start() is fully non-blocking).

Sequence Tracker — Loss Detection

The SequenceTracker uses a sliding-window bitmap (default 4096 entries). On each RecordRecv(seq) it marks the sequence number seen. Flush(lastSentSeq) at end-of-run counts unmarked positions as lost.

⚠ Combined Transport Limitation

When running with --receivers N, all N clients share the same recorder. The server broadcasts seq=42 to all N clients; all N call RecordRecv(42). The SeqTracker sees seq=42 N times → reports N−1 duplicates per message.

What is still accurate: Latency (HDR histogram records each receive timestamp independently). What is inflated: MsgPerSec (= N × actual per-receiver rate), Duplicated count. Divide MsgsPerSec by ReceiverCount for per-receiver throughput.

Known Trade-offs & Limitations

Fixed

TCP multi-listener crash on Windows

SO_REUSEPORT is Linux-only. Without it, binding N listeners to the same port fails on the 2nd call (WSAEADDRINUSE). Fixed: if runtime.GOOS != "linux" { n = 1 } in TCPServer.Start(). Single listener is used on Windows/macOS.
Fixed

Async stream registration race (HTTP/1, HTTP/2, SSE, HTTP/3)

HTTP/1, HTTP/2, SSE clients start their stream goroutine asynchronously — the server-side handler fires after Connect() returns. HTTP/3 Start() is fully async. Old code used guessed fixed sleeps. Fixed: waitForConns(server, N) polls Connections() every 10ms with 10–15s timeout, guaranteeing broadcasts only start after all clients are registered.
Fixed

Wire encoding inconsistency (8 servers)

HTTP/3 and TCP/UDP already called wire.Encode() in Broadcast(). The other 8 servers (HTTP/1, HTTP/2, SSE, WebSocket×3, WebRTC, WebTransport) sent raw generator bytes. Fixed: all 8 servers now call wire.Encode(seqCounter.Add(1), time.Now().UnixNano(), data) before sending, enabling latency measurement for all 11 protocols.
Architectural

N-receiver seq inflation in combined transport

All N embedded clients share one recorder. Every broadcast's seq number is recorded N times → (N−1)×MsgCount duplicates counted; SeqTracker loss detection unreliable. MsgPerSec is aggregate (N× actual per-receiver rate).
Workaround: For accurate per-receiver metrics, use standalone cmd/xxx-server + multiple cmd/xxx-client processes. For throughput comparison, divide MsgsPerSec by --receivers. For latency comparison, the HDR histogram is correct regardless.
Known

benchmark.yaml config not wired to cobra flags

Viper reads benchmark.yaml at startup but viper.BindPFlag() is never called. Flag defaults are not populated from the YAML. The file is effectively documentation-only at this point.
Workaround: Pass all parameters explicitly as CLI flags. The YAML serves as a configuration template showing available options.
Known

WebRTC uses Google STUN server

WebRTCClient.Connect() includes stun:stun.l.google.com:19302 in the ICE configuration. On machines without internet access, ICE gathering still produces loopback candidates (works locally) but may time out waiting for STUN reflexive candidates. The 15s waitForConns timeout accommodates this.
Not implemented

Scenario logic (A–E)

Scenarios A–D are thin wrappers that set ReceiverCount and call NewRunner(cfg).Run(ctx). They don't implement the different subscriber behaviour described in their comments (e.g. Scenario B: 100 subscribers each subscribing to a 10-instrument subset). Scenario E falls back to 1000-receiver local mode (Kubernetes path returns an error).
Workaround: Use --receivers N + --duration Xs to emulate scenarios manually. Selective subscription requires the TopicTransport interface which is defined but not wired to the scenario runner.
Not implemented

Broadcast strategy selection

internal/broadcast/ has four strategy implementations: NaiveBroadcaster, WorkerPoolBroadcaster, ShardedBroadcaster, EpollBroadcaster. The --broadcast-strat flag is accepted but the scenario runner never uses it — broadcastLoop always calls srv.Broadcast() directly, bypassing this selection entirely.
To fix: The runner needs a Broadcaster wrapper that intercepts Broadcast(data), routes through the chosen strategy, and forwards each message to the underlying transport's per-connection send paths.
